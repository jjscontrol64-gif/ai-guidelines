# 신뢰성 기반 통찰 탐색 지침 (Reliable Insight Exploration)

## 목적
개발 관련 결정·주장의 근거를 찾을 때, **출처의 신뢰도 계층**과 **표현의 직접성**을 기준으로 탐색하여 추측이 아닌 검증된 통찰을 얻는다. (AI 에이전트뿐 아니라 모든 기술 리서치에 적용 가능)

## 출처 신뢰도 계층 (높음 → 낮음)
1. **1차 출처 / 공식 문서** — 도구·기술 제공사의 공식 가이드, 사양, API 레퍼런스, 표준 문서. 가장 우선하며 모든 주장의 기준점.
2. **실제 제품·운영 문서** — 출시된 제품에 실제 반영된 기능·워크플로 문서(예: 특정 동작 모드의 작동 방식). "이론"이 아니라 "실제 구현"을 보여주므로 높게 평가.
3. **연구 논문 / 벤치마크** — 가설을 실험·측정으로 검증한 자료. 인과·효과 주장의 근거. 재현성·표본·실험 조건을 함께 확인.
4. **전문 / 교육 커뮤니티** — 실무자 포럼, 공식 디스커션 등. 빠른 단서·사례 수집엔 유용하나 공식 문서보다 신뢰도 낮음. 결론이 아니라 "가설·리드"로만 사용.

> **공식 연구·기술 블로그 취급**: 제공사 리서치 블로그는 성격에 따라 분리 평가한다. 자사 제품·기능에 대한 1차 진술이면 계층 1에 준해 신뢰. 일반적·실험적 주장이면 동료평가 전이므로 계층 3 미만(가설)로 보고 논문·벤치마크로 보강. 홍보·발표성 표현과 기술적 검증 내용을 구분할 것.

## 탐색 원칙
- **상위 우선**: 상위 계층에서 먼저 확인하고, 하위 계층은 보강·반례·실무 맥락 확인용으로만 사용.
- **교차 검증**: 핵심 주장은 최소 2개 이상의 독립 출처로 확인. 충돌 시 상위 계층·최신 자료를 우선.
- **표현의 직접성 판별**: 출처가 주장을 직접 진술했는지, 의미가 유사·유추되는지 구분해 표기.
  - `[직접]` 원문이 해당 내용을 명시적으로 서술함.
  - `[유추]` 원문에서 추론했거나 인접 의미로 해석함.
  - 직접 진술을 유추보다 높게 신뢰하며, 유추는 반드시 추가 근거로 보강.
- **직접성 ≠ 근거 강도**: 출처가 직접 진술해도 근거(실험·데이터)가 약할 수 있음. 특히 블로그·커뮤니티는 직접성과 별개로 근거 강도를 따로 평가.
- **최신성·버전 추적**: 문서의 "최종 수정일"·버전·변경 이력(changelog/release notes)을 확인. 현재 stable 버전 문서를 우선하고 deprecated 항목은 배제. 학습 데이터(기억)에 의존하지 말고 최신 원문을 직접 조회.
- **출처 표기**: 결론마다 출처·계층(1~4)·직접성 태그를 남겨 검증 가능하게 함.

## 권장 워크플로
1. 질문을 검증 가능한 **단일 주장**들로 분해.
2. 각 주장을 계층 1 → 2 → 3 → 4 순으로 탐색.
3. 상위에서 직접 근거를 찾으면 확정, 없으면 하위로 내려가며 "가설"로 표시.
4. 직접/유추·출처·최신성을 태깅해 정리.
5. 충돌·공백이 있으면 추가 탐색하거나 "불확실"로 명시.

## 산출물 형식
각 결론은 다음 형태로 정리한다:
> **주장** — 근거 출처(계층 N) — `[직접]`/`[유추]` — 신뢰도(상/중/하)
